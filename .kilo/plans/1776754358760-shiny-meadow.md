# send-lxmf Daemon Design

## Goal

Create a system-wide LXMF message daemon that owns the router, identity, and storage. CLI tools become lightweight clients that submit messages via Unix socket.

## Architecture

```
┌─────────────────────────────────────────────────────┐
│              send-lxmf-daemon                       │
│                                                     │
│  - Owns identity file                               │
│  - Owns LXMRouter with jobloop                      │
│  - Owns storage directory                           │
│  - Listens on Unix socket (/run/send-lxmf/sock)     │
│  - Handles all peer sync & message routing          │
└─────────────────────────────────────────────────────┘
                    ▲
                    │ Unix socket (IPC)
                    │
    ┌───────────────┼───────────────┐
    │               │               │
    ▼               ▼               ▼
┌──────────┐  ┌──────────────┐  ┌────────┐
│send-lxmf │  │sendmail-lxmf │  │ lxmd   │
│(client)  │  │  (client)    │  │        │
└──────────┘  └──────────────┘  └────────┘
```

## Components

### 1. Daemon (`send-lxmf-daemon`)

**Storage locations:**
- Identity: `{data_dir}/identity`
- LXMRouter storage: `{data_dir}/lxmf/`
- Socket: `/run/send-lxmf/send-lxmf.sock` (or `XDG_RUNTIME_DIR/send-lxmf/send-lxmf.sock`)
- PID file: `/run/send-lxmf/daemon.pid`

**Configuration** (from `/etc/send-lxmf.conf`):
- `data-dir` - storage directory
- `identity` - path to identity file (overrides default)
- `propagation-node` - fallback propagation node
- `display-name` - sender display name

**Responsibilities:**
- Initialize Reticulum (if not already running)
- Load/create identity
- Create LXMRouter with storage path
- Start jobloop in background thread
- Listen on Unix socket for incoming message requests
- Handle message submission, delivery callbacks, status queries

**Message protocol** (via socket):
- Client sends JSON message: `{"action": "send", "destinations": [...], "content": "...", "title": "...", ...}`
- Daemon responds: `{"status": "queued", "message_id": "..."}` or `{"status": "error", "error": "..."}`

### 2. Client Library (shared code in `lib.py`)

**Functionality:**
- Connect to daemon socket
- Submit message request
- Wait for response / delivery confirmation
- Handle retries and failures

**Replaces current `send_message()` function behavior** but delegates to daemon instead of creating own router.

### 3. CLI Tools

**`send-lxmf`:**
- Reads config, parses args
- Connects to daemon socket
- Sends message request
- Prints status / delivery confirmation
- Exits (no own router)

**`sendmail-lxmf`:**
- Same pattern - reads email from stdin, submits to daemon

### 4. Service Management

**systemd integration:**
- `send-lxmf-daemon.service` - starts daemon, always-on (no socket activation)
- Runs as dedicated user `reticulum`
- Dependencies: `rnsd.service` (Reticulum transport)
- No propagation node functionality (separate from lxmd)

## Implementation Plan

### Phase 1: Core Daemon
1. Create `daemon.py` with:
   - `LXMDaemon` class managing router lifecycle
   - Unix socket server using stdlib `socket` module
   - JSON message protocol
   - Signal handling (SIGTERM, SIGHUP)
   - PID file management

2. Create `client.py` with:
   - `DaemonClient` class
   - Socket connection with timeout
   - Request/response handling
   - Retry logic on connection failure

### Phase 2: Update lib.py
1. Add `send_message_via_daemon()` function
2. Preserve `send_message()` for standalone mode (when daemon unavailable)
3. Auto-detect: if `daemon-socket` config is set, connect to daemon; otherwise use direct mode

### Phase 3: CLI Updates
1. `send.py` - use daemon client instead of direct send_message
2. `sendmail.py` - use daemon client instead of direct send_message

### Phase 4: System Integration
1. Create `send-lxmf-daemon` entry point in pyproject.toml
2. Add systemd service file
3. Add init script for non-systemd systems

## Configuration Options

```ini
[send-lxmf]
default-destination = ...
propagation-node = ...
display-name = ...
data-dir = /var/lib/send-lxmf
identity = /var/lib/send-lxmf/identity

# Daemon mode
daemon-socket = /run/send-lxmf/send-lxmf.sock
```

## Message Protocol

### Client → Daemon

```json
{
  "action": "send",
  "id": "uuid-123",
  "destinations": ["b9af7034186731b9f009d06795172a36"],
  "content": "Hello world",
  "title": "Test message",
  "display_name": "My Name",
  "prepend_title": true,
  "attachments": ["/path/to/file"],
  "propagation_node": "optional-pn-hash",
  "timeout": 30
}
```

### Daemon → Client

```json
{
  "status": "queued",
  "message_id": "uuid-123"
}
```

Or on error:
```json
{
  "status": "error",
  "error": "Failed to queue message"
}
```

## Backward Compatibility

- If `daemon-socket` is not configured or daemon is not running, fall back to direct mode (current behavior)
- `send-lxmf --direct` flag forces direct mode
- Log warning when falling back to direct mode

## Design Decisions

1. **Daemon user**: `reticulum` (dedicated user, not nobody)
2. **lxmd relationship**: Send-lxmf daemon runs separately from lxmd propagation node, they share Reticulum transport via rnsd
3. **Delivery notifications**: Log only, no callback to clients
4. **Daemon mode**: Always-on, no socket activation (simple approach)
